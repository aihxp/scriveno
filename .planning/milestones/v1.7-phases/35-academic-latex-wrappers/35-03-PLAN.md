<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| writer -> --platform flag | Platform slug validated against hard-coded allowed list in STEP 2.5; unrecognised values rejected before any tool is called |
| STEP 2 -> kpsewhich subprocess | kpsewhich is a read-only TeX path query; no user-supplied input is passed to it; class file argument is derived from the validated platform slug via a fixed map |
| STEP 4 -> pandoc subprocess | Template path is derived from the validated platform slug via a fixed pattern; no arbitrary file path from writer input reaches the --template flag |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-04 | Tampering | --platform flag input | mitigate | STEP 2.5 validates slug against an explicit allowed-values list before any subprocess call; invalid slugs produce an error and stop |
| T-35-05 | Spoofing | kpsewhich path | accept | kpsewhich is invoked as `command -v kpsewhich` - PATH-resolved, no hardcoded path; same pattern used for pandoc/typst/gs throughout the command |
| T-35-06 | Denial of Service | pdflatex compilation loop | accept | Scriveno does not invoke pdflatex; writer controls their own TeX workflow |
| T-35-07 | Information Disclosure | .tex file contents | accept | .tex file written to writer's own .manuscript/output/; no network transmission |
</threat_model>
